Ты выступаешь в роли senior software architect + senior full-stack desktop engineer + EDA/simulation systems engineer.

Нужно спроектировать и начать реализацию десктопной инженерной программы для расчёта, симуляции и документирования электрических схем.

Название проекта условное: HotSAS Studio (Hot Schematic Analysis & Simulation Studio)


Главная цель:
Создать модульную десктопную программу для схемотехники, где пользователь может:
- собирать электрические схемы в удобном UI;
- использовать встроенный инженерный калькулятор;
- работать с библиотекой формул;
- создавать схемные заготовки из формул;
- подбирать номиналы по E-рядам E12/E24/E48/E96/E192;
- использовать реальные компоненты с параметрами, SPICE-моделями, symbol и footprint;
- запускать SPICE-симуляцию;
- смотреть графики токов, напряжений, фаз, усиления, импедансов;
- анализировать выделенный участок цепи с указанием входа и выхода;
- экспортировать расчёты, отчёты, BOM, netlist, схемы и библиотеки;
- в будущем экспортировать symbol+footprint данные для удобной работы с Altium/KiCad.

ВАЖНО:
На первом этапе НЕ нужно реализовывать PCB-редактор.
Но архитектурно нужно заложить поддержку symbol+footprint, чтобы в будущем можно было экспортировать библиотеки посадочных мест и данные компонентов для EDA-систем.

- отправляет команды в backend/application layer;
- показывает DTO, полученные от backend.

Запрещено размещать в UI:
- расчёт E-рядов;
- генерацию SPICE netlist;

- прямой доступ к SQLite.

Вся предметная логика должна быть в Rust backend/core/application services.

============================================================
2. Главный архитектурный стиль
============================================================

Использовать Clean Architecture / Hexagonal Architecture / Ports and Adapters.

Слои:

1. UI Layer
   - Tauri + React + TypeScript.
   - Только отображение и пользовательский ввод.

2. Application Layer
   - use cases / application services.
   - Команды: createProject, addComponent, connectPins, updateParameter, runSimulation, exportReport, importModel и т.д.

3. Domain Core
   - чистая предметная модель:
     Circuit, Component, Net, Wire, Formula, PreferredValueSeries, SimulationProfile, Report, Symbol, Footprint.

4. Ports
   - интерфейсы к внешним системам:
     StoragePort, SimulationEnginePort, ExporterPort, ImporterPort, ModelParserPort, FormulaEnginePort.

5. Adapters
   - конкретные реализации:
     SQLiteStorageAdapter, NgspiceSimulationAdapter, SpiceNetlistExporter, PdfReportExporter, KicadExporter и т.д.

Backend/core не должен знать о React, Tauri, HTML, CSS, Qt, Avalonia, egui и т.д.

8. Инженерный калькулятор
============================================================

Нужно заложить калькулятор как Engineering Notebook.

Калькулятор должен поддерживать:
- обычный ввод выражений;
- инженерные приставки: k, M, m, u, n, p;
- единицы измерения;
- комплексные числа;
- dB, dBm;
- импедансы;
- переменные;
- историю;
- прямую запись формул;
- LaTeX-like ввод;
- Wolfram-like команды;
- графики;
- связь результата с компонентом схемы;
- добавление расчёта в отчёт;
- пользовательские формулы.

Режимы ввода:

1. Expression mode:
R = 10k
C = 100n
fc = 1 / (2*pi*R*C)

2. LaTeX-like mode:
f_c = \frac{1}{2\pi R C}
R = 10\,k\Omega
C = 100\,nF

3. Wolfram-like command mode:
solve(fc = 1/(2*pi*R*C), R)
nearestE(15.93k, E24)
sweep(R, E24, 1k..100k, fc = 1/(2*pi*R*C))
bode(H = 1/(1+s*R*C), f=10..1e6)

На первом этапе можно сделать:
- parser interface;
- заглушку/простую реализацию expression evaluator;
- сохранение notebook blocks;
- UI-панель ввода;
- отображение результата;
- историю расчётов.

Позже можно подключить:
- math.js;
- SymPy bridge;
- Lcapy bridge.

ВАЖНО:
Даже если вычислительный движок сначала простой, архитектурно нужен FormulaEnginePort.

============================================================
9. Библиотека формул
============================================================

Формулы должны быть расширяемыми пользователем.

Формулы хранить не только в коде, а в JSON/YAML formula packs.

Пример YAML-формулы:

id: rc_low_pass_cutoff
title: RC Low-Pass Cutoff Frequency
category: filters/passive
description: Cutoff frequency of first-order RC low-pass filter

variables:
  R:
    unit: Ohm
    description: Resistance
    default: 10k
  C:
    unit: F
    description: Capacitance
    default: 100n

equations:
  - id: cutoff
    latex: "f_c = \\frac{1}{2\\pi R C}"
    expression: "fc = 1 / (2*pi*R*C)"
    solve_for: ["fc", "R", "C"]

outputs:
  fc:
    unit: Hz

linkedCircuitTemplateId: rc_low_pass_template

mapping:
  R: R1.resistance
  C: C1.capacitance
  Vin: net_in
  Vout: net_out

defaultSimulation:
  type: ac_sweep
  start: 10 Hz
  stop: 1 MHz
  pointsPerDecade: 100

Формула должна быть связана со схемной заготовкой.
Пользователь должен иметь возможность:
- открыть формулу;
- ввести параметры;
- рассчитать;
- подобрать E-ряды;
- создать схемную заготовку;
- запустить симуляцию;
- добавить расчёт в отчёт.

============================================================
12. Анализ выделенного участка цепи
============================================================

Нужно заложить возможность расчёта только выделенного участка схемы.

Пользовательский сценарий:
1. Пользователь выделяет часть схемы.
2. Нажимает Analyze Selection.
3. Программа подсвечивает внешние узлы.
4. Пользователь выбирает:
   - input+;
   - input- или GND;
   - output+;
   - output- или GND;
   - reference node;
   - positive analysis direction.
5. Программа предлагает режим:
   - symbolic linear analysis;
   - numeric SPICE analysis;
   - template-based explanation;
   - all available.

============================================================

Formulas must be user-extensible.

Do not store formulas only in code. Use JSON/YAML formula packs.

Example YAML formula:

id: rc_low_pass_cutoff
title: RC Low-Pass Cutoff Frequency
category: filters/passive
description: Cutoff frequency of first-order RC low-pass filter

variables:
  R:
    unit: Ohm
    description: Resistance
    default: 10k
  C:
    unit: F
    description: Capacitance
    default: 100n

equations:
  - id: cutoff
    latex: "f_c = \\frac{1}{2\\pi R C}"
    expression: "fc = 1 / (2*pi*R*C)"
    solve_for: ["fc", "R", "C"]

outputs:
  fc:
    unit: Hz

linkedCircuitTemplateId: rc_low_pass_template

mapping:
  R: R1.resistance
  C: C1.capacitance
  Vin: net_in
  Vout: net_out

defaultSimulation:
  type: ac_sweep
  start: 10 Hz
  stop: 1 MHz

============================================================
12. Selected circuit region analysis
============================================================

The application must support analysis of only a selected region of the circuit.

User scenario:
1. User selects a region of the schematic.
2. User clicks Analyze Selection.
3. The app highlights boundary nodes.
4. User selects:
   - input+;
   - input- or GND;
   - output+;
   - output- or GND;
   - reference node;
   - positive analysis direction.
5. The app offers analysis modes:
   - symbolic linear analysis;
   - numeric SPICE analysis;
   - template-based explanation;
   - all available.
6. The app outputs:
   - formulas;
   - intermediate calculations;
   - equivalent circuit;
   - transfer function if available;
   - graphs;
   - warnings;
   - SPICE netlist fragment;
   - option to add the analysis to a report.

Important:
Do not promise symbolic formulas for arbitrary circuits.
Rules:
- if the region is linear, output H(s), Zin, Zout, poles/zeros, Bode;
- if the region matches a known template, output standard engineering formulas;
- if the region is nonlinear, output numerical SPICE results and approximate formulas;
- if the region is too complex, output graphs, netlist, and a warning that symbolic derivation is unavailable.

Required module:
SelectedRegionAnalysisService

DTO:
SelectedCircuitRegion
- component_ids
- internal_nets
- boundary_nets
- input_port
- output_port
- reference_node
- analysis_direction
- analysis_mode

Graphs for selected region:
- Vin(t), Vout(t);
- Iin(t), Iout(t);
- gain;
- phase;
- Zin(f);
- Zout(f);
- power;
- ripple;
- group delay.

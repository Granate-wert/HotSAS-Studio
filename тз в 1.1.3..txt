# 4. Найденные проблемы и недочёты после v1.1.2

## 4.1. Форматирование всё ещё выглядит проблемным

Многие raw-файлы всё ещё отображаются как одна длинная строка. Например `engine/Cargo.toml` отображается одной строкой, `package.json` тоже одной строкой, а `docs/testing/TESTING.md` занимает всего 4 строки с большим количеством Markdown внутри одной строки. ([GitHub][9])

Это особенно плохо для `docs/testing/TESTING.md`: документ должен был стать читаемым Markdown, но сейчас GitHub raw показывает его как сжатый текст, где заголовки, абзацы и code blocks идут почти подряд. ([GitHub][9])

**Вывод:** перед v1.1.3 нужно отдельным preflight-блоком исправить форматирование Markdown/YAML/TOML/JSON, если локально файлы действительно выглядят так же.

---

## 4.2. HTML report escaping выглядит подозрительно

В `MarkdownReportExporter::export_html` raw-представление показывает цепочку:

```rust
.replace('&', "&")
.replace('<', "<")
.replace('>', ">")
```

То есть по raw-виду символы не превращаются в `&amp;`, `&lt;`, `&gt;`. Возможно, web-инструмент декодирует HTML entities при отображении, но тесты и реализацию нужно перепроверить локально. ([GitHub][10])

Ещё подозрительнее выглядит `report_export_tests.rs`: raw-вывод показывает проверки, которые выглядят как будто они ожидают `<script>alert(1)</script>` в HTML, а это противоположно нашему требованию безопасного escaping. С учётом особенностей отображения raw через инструмент я не могу на 100% утверждать, что исходник именно такой, но это обязательно нужно перепроверить в v1.1.3 preflight. ([GitHub][11])

**Вывод:** v1.1.3 должна начинаться с проверки и исправления HTML escaping tests/exporter. Это не новая функция, а исправление качества/безопасности.

---

## 4.3. Formula packs выглядят неготовыми для runtime-загрузки

`shared/formula_packs/filters.yaml` в raw-виде выглядит как 3 строки, где `packId`, `title`, `version`, `formulas`, `variables`, `equations`, `outputs`, `mapping`, `defaultSimulation` идут в одной строке. Такой файл, скорее всего, будет неудобен или вовсе невалиден для нормального YAML-парсера. ([GitHub][12])

То же самое видно в `basic_electronics.yaml`: формулы `ohms_law` и `voltage_divider` находятся в сильно сжатом виде. ([GitHub][13])

**Вывод:** в v1.1.3 нельзя просто написать `serde_yaml::from_str` и ожидать, что всё заработает. Сначала нужно нормализовать formula packs в валидный многострочный YAML.

---

## 4.4. Формулы всё ещё hardcoded в Rust

В `templates.rs` функция `rc_low_pass_formula()` всё ещё создаёт `FormulaDefinition` прямо в Rust-коде. Это допустимо до v1.1.2, но v1.1.3 должна сделать `shared/formula_packs/filters.yaml` источником истины для Formula Library. ([GitHub][14])

В `models.rs` уже есть `FormulaDefinition`, `FormulaEquation`, `FormulaVariable`, `FormulaOutput`, `CircuitTemplate`, `FormulaCircuitBinding`. Это хорошая основа для v1.1.3, но пока нет `FormulaPack`, `FormulaPackLoader`, `FormulaRegistry`. ([GitHub][15])

---

## 4.5. FormulaEnginePort всё ещё RC-specific

В `formula_engine_tests.rs` используется только `calculate_rc_low_pass_cutoff`, и это соответствует текущему `FormulaEnginePort`, но для v1.1.3/1.1.4 это уже ограничение. ([GitHub][16])

Для v1.1.3 ещё не нужно делать полноценный generic FormulaEngine, но нужно подготовить registry так, чтобы v1.1.4 смог заменить RC-specific вычисление на `evaluate_formula(formula_id, variables)`.

---

## 4.6. E48/E96/E192 пока остаются приближёнными

`preferred_values.rs` всё ещё использует точные таблицы только для E3/E6/E12/E24, а для остальных рядов применяет `generated_base_values(other.count())`. Это было допустимо для v1.1.2, потому что точные таблицы мы вынесли в v1.1.5. ([GitHub][17])

**Вывод:** не трогать в v1.1.3, но не забывать как технический долг.

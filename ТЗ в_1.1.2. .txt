# 6. Test helper для сравнения f64

Добавить helper там, где удобно: либо в каждом тестовом файле, либо в test utility module.

Пример:

```rust id="zluj0p"
fn assert_approx_eq(actual: f64, expected: f64, epsilon: f64) {
    assert!(
        (actual - expected).abs() <= epsilon,
        "actual={actual}, expected={expected}, epsilon={epsilon}"
    );
}
```

Использовать для:

```text id="bbp4x6"
- fc;
- parsed engineering values;
- preferred value calculations;
- percent error;
- graph/mock simulation values, если нужно.
```

Не сравнивать floating point через `==`, кроме случаев, где это абсолютно безопасно.

# 13. Тесты HTML Report Export

## 13.1. Цель

Проверить, что HTML export безопасно экранирует пользовательский/отчётный текст.

## 13.2. Critical escaping test

Создать report section или markdown body с содержимым:

```html id="mitqpr"
<script>alert(1)</script>
```

HTML output не должен содержать активный тег:

```html id="wy79c7"
<script>alert(1)</script>
```

HTML output должен содержать escaped text:

```html id="i5uayd"
&lt;script&gt;alert(1)&lt;/script&gt;
```

## 13.3. Проверить символы

Добавить тесты:

```text id="n8exib"
"&" -> "&amp;"
"<" -> "&lt;"
">" -> "&gt;"
```

Порядок escaping важен:

```text id="t6sv5k"
1. Сначала заменить & на &amp;
2. Потом < на &lt;
3. Потом > на &gt;
```

Если сделать наоборот, можно получить двойное или некорректное экранирование.

## 13.4. Минимальное исправление exporter

Если текущий `export_html` неправильный, исправить минимально.

Пример допустимой реализации:

```rust id="ktv0ut"
fn escape_html(input: &str) -> String {
    input
        .replace('&', "&amp;")
        .replace('<', "&lt;")
        .replace('>', "&gt;")
}
```

Не добавлять полноценный Markdown-to-HTML renderer.
На v1.1.2 достаточно безопасного `<pre>` или простого HTML wrapper.
